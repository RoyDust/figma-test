## 上下文

当前项目只有 OpenSpec 和 Codex 相关配置，还没有实际应用代码。目标是从零搭建一个真实可用的 Twitter 风格 MVP，并优先保证本地开发易启动、功能闭环完整、后续可扩展。

第一版技术约束已经明确：

- 使用 Next.js App Router 构建前后端一体化应用
- 使用 Prisma 管理数据模型和数据库访问
- 使用本地 SQLite 降低环境搭建成本
- 使用 NextAuth 处理认证和会话

## 目标 / 非目标

**目标：**

- 支持用户注册、登录、退出
- 支持已登录用户发布短帖
- 支持首页时间线按时间倒序展示帖子
- 支持已登录用户点赞和取消点赞
- 支持本地环境快速启动和测试

**非目标：**

- 不实现关注关系
- 不实现评论、转发、通知
- 不实现图片上传或富文本内容
- 不实现推荐算法或实时通信
- 不为了首版引入复杂的分布式服务拆分

## 决策

### 1. 使用 Next.js App Router 作为单仓全栈架构

- 原因：页面、接口和服务端渲染可以在同一应用内完成，适合从零构建 MVP，开发路径更短
- 备选方案：
  - Next.js Pages Router：更传统，但对新项目不如 App Router 一致
  - React + Express 分离：边界更清晰，但首版会增加项目结构和通信成本

### 2. 使用 Prisma + SQLite 作为首版数据层

- 原因：SQLite 在本地开发几乎零额外运维成本，Prisma 能保留后续迁移到 PostgreSQL 的灵活性
- 备选方案：
  - PostgreSQL：更接近生产，但本地初始化更重
  - 直接使用原生 SQL：控制力更强，但首版开发效率更低

### 3. 使用 NextAuth 管理认证和会话

- 原因：减少手写认证流程的出错面，便于快速建立稳定的登录态与受保护操作
- 备选方案：
  - 自研 session 体系：灵活，但需要额外处理密码、cookie 和会话安全细节
  - 更换其他认证库：会引入额外选型成本，不适合当前 MVP

### 4. 将能力拆分为认证、发帖、时间线、点赞四个规范模块

- 原因：能力边界清晰，便于分别编写需求、任务与后续增量变更
- 备选方案：
  - 单一“大而全”规范：起步快，但后续维护和扩展不清晰

## 风险 / 权衡

- SQLite 不适合高并发生产场景 -> 首版仅作为本地与 MVP 数据层，后续可借助 Prisma 迁移
- NextAuth 与 App Router 集成细节较多 -> 通过限制首版登录方式和测试关键流程降低复杂度
- 首版没有关注关系，时间线会是全局流 -> 接受为 MVP 权衡，后续通过新 capability 扩展
- 从零搭建全栈项目涉及多个模块 -> 通过 OpenSpec 任务拆分和测试优先策略控制风险

## Migration Plan

- 当前仓库没有现有应用代码，因此不存在旧系统迁移成本
- 第一步搭建 Next.js 与 Prisma 基础骨架
- 第二步创建数据库 schema 和认证基础设施
- 第三步实现帖子、时间线、点赞闭环
- 第四步补充测试并验证本地启动流程
- 如果中途方案不成立，可回退到仅保留已验证通过的最小功能集
